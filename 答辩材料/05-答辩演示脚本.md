# 答辩演示脚本

## 演示流程总览（建议 15-20 分钟）

```
1. 项目概述与架构 ──── 3 分钟
2. Web 读者端演示 ──── 5 分钟
3. Web 管理端演示 ──── 4 分钟
4. 移动端演示 ──────── 3 分钟
5. 技术亮点总结 ──── 3 分钟
```

---

## 第一部分：项目概述与架构（3 分钟）

### 演示内容

1. 展示系统架构图（`01-系统架构设计.md` 中的架构图）
2. 说明 **"统一后端 + 多端读者入口 + Web 管理后台"** 的架构特点
3. 展示 E-R 关系图（`02-数据库设计.md` 中的 mermaid 图）
4. 快速展示 **核心业务级联关系**（`03-业务状态机.md` 中的总图）

### 话术参考

> 本系统采用前后端分离架构，后端基于 Spring Boot 提供统一的 REST API，
> 前端通过 Web、Android 和微信小程序三端接入同一套 API。
> 系统涵盖借阅、预约、罚款、通知、反馈等完整业务闭环。

---

## 第二部分：Web 读者端演示（5 分钟）

### 演示路径

```
步骤 1：打开首页 → 展示首页聚合效果
        └─ 推荐图书、新书推荐、分类快捷入口

步骤 2：进入馆藏目录 → 搜索 "Python"
        └─ 展示联想词、热门搜索、搜索历史
        └─ 展示分类筛选和库存筛选

步骤 3：点击一本图书 → 进入详情页
        └─ 展示书籍信息、馆藏位置、评论区

步骤 4：发起借阅或预约
        └─ 有副本 → 点"借阅" → 系统分配副本
        └─ 无副本 → 点"预约" → 进入排队

步骤 5：进入个人中心
        └─ 展示：书架 / 借阅追踪 / 预约 / 罚款 / 通知

步骤 6：查看通知中心
        └─ 展示：通知列表、标记已读、深链跳转到具体页面
```

### 要点

- 强调**搜索发现体验**（联想词 + 热词 + 历史）
- 强调**通知深链跳转**（点击通知能直接跳到对应的借阅/预约/罚款详情）
- 演示完美体验的**加载 → 数据 → 空状态 → 错误**四种 UI 状态

## 第三部分：Web 管理端演示（4 分钟）

### 演示路径

```
步骤 1：登录管理员账号 → 进入仪表盘
        └─ 展示：馆藏总量、借阅数、逾期数、预约数

步骤 2：进入图书管理 → 新增一本图书
        └─ 展示：支持实体/线上/混合三种资源模式

步骤 3：进入借阅管理 → 确认还书
        └─ 展示：状态筛选、确认还书、标记遗失

步骤 4：进入罚款管理
        └─ 展示：分页搜索、状态 Tab、待缴总额
        └─ 操作：柜台收款 或 豁免

步骤 5：进入 RBAC 设置（如有时间）
        └─ 展示：角色管理 → 权限分配 → 审计日志
```

### 要点

- 强调**业务规则自动化**（逾期自动罚款、遗失自动赔偿）
- 强调**细粒度权限控制**（`loan:manage`、`fine:waive` 等权限粒度）

## 第五部分：技术亮点总结（3 分钟）

### 推荐展示的亮点

1. **业务闭环**：借阅 → 预约 → 罚款 → 通知的完整级联
2. **统一 API 多端复用**：三端共享 29 个 Controller、80+ 个接口
3. **搜索发现体验**：热词 + 联想 + 历史，三个端都落地
4. **RBAC 权限模型**：角色→权限→API→页面的完整控制链
5. **状态机设计**：借阅、预约、罚款多层状态流转自动触发
6. **通知深链**：通知点击可精准跳转到对应业务页面

## 常见答辩问题准备

| 可能的问题 | 参考回答 |
|---|---|
| 系统有哪些业务表？ | 28 张业务表，覆盖用户、图书、副本、借阅、预约、罚款、通知、推荐、座位、服务预约、RBAC 等 |
| 如何保证借阅数据一致性？ | Spring 事务 + 乐观锁 + 业务校验三重保障 |
| 移动端和 Web 有什么不同？ | 共享同一套 API，移动端聚焦读者场景，Web 额外提供管理端 |
| 预约排队怎么实现的？ | 按预约时间排序，归还时自动分配给队首用户 |
| 权限怎么控制的？ | JWT + RBAC，角色→权限→@PreAuthorize 方法级控制 |
| 为什么用三种前端技术栈？ | 展示多端开发能力：Web SSR、移动跨平台、微信生态原生 |
| 逾期罚款怎么生成的？ | 定时任务检测逾期，归还时自动计算并生成罚款+通知 |
| 有没有做测试？ | 51 个测试文件，356 个测试用例，通过率 96.6%，对核心链路做了端到端验证 |
| 为什么有少量测试失败？ | 12 个失败均为测试代码未同步业务代码变更（中文化消息、新增参数），不影响实际功能 |
| 系统的创新点在哪？ | 四端协同、AI 助手、座位预约、社交推荐、服务预约、通知深链跳转 |
