# 图书馆资源查询与借阅系统测试模块（论文草稿）

## 1 测试目标

为验证图书馆资源查询与借阅系统在实际运行中的可用性、正确性与稳定性，本文从接口层、业务层和联调层三个维度对系统进行测试。测试工作的核心目标如下：

1. 验证后端接口能够按照既定的 RESTful 规范正确响应请求。
2. 验证系统在认证、授权、参数校验、业务状态流转等方面的逻辑正确性。
3. 验证借阅、预约、罚款、通知等核心业务流程能够形成完整闭环。
4. 验证系统后端接口能够稳定支撑 Web 端、Android 端和小程序端的调用需求。

本系统并非单纯的信息展示系统，而是包含借阅管理、预约排队、罚款处理、消息通知和角色权限控制等业务规则，因此测试重点不仅包括接口可访问性，还包括权限限制与业务状态变化是否符合预期。

## 3 测试方法与方案

### 3.1 测试思路

结合本系统的实际架构，本文采用“自动化接口测试为主、人工联调验证为辅”的测试方案。

1. 控制层接口测试  
   使用 `@WebMvcTest`、`MockMvc` 对认证、借阅、预约、通知、收藏等接口进行测试，重点验证接口状态码、返回数据结构、权限控制和异常响应。

2. 业务集成测试  
   使用 `@SpringBootTest`、`@AutoConfigureMockMvc` 和 H2 内存数据库验证需要真实数据状态变化的业务流程，例如罚款查询、预约履约、服务预约等接口。

3. 多端联调验证  
   结合 Web 端、Android 端和小程序端，对登录、图书查询、借阅记录、通知读取等典型功能进行联调，验证前后端接口契约一致，确保系统具备实际部署基础。

### 3.2 方案选择原因

若论文仅采用 Postman 或 Apifox 手工调用接口，虽然能够演示请求结果，但很难系统性覆盖权限分支、异常输入和业务状态变化，也不利于后期回归测试。相比之下，本系统已有较完善的后端自动化测试基础，更适合作为论文测试模块的核心支撑。

从项目现状看，后端已经具备较成熟的测试代码：

- 控制层测试 24 个
- 服务层测试 21 个
- 安全相关测试 3 个

这些测试覆盖了认证、图书、借阅、预约、罚款、评论、用户、角色权限、通知等关键模块，能够较真实地反映系统质量情况。因此，本文最终选择以后端自动化接口测试为主线，将人工联调用作补充说明，这也是当前项目条件下最合理、最有说服力的方案。

## 4 测试内容设计

### 4.1 测试重点

根据系统业务特点，本文将接口测试重点划分为以下几类：

1. 认证与身份识别测试  
   验证注册、登录、获取当前用户信息、退出登录、刷新令牌等接口是否正常工作。

2. 权限与角色控制测试  
   验证普通用户、管理员以及具备特定权限的操作员访问不同接口时是否能够被正确放行或拦截。

3. 核心业务流程测试  
   验证借书、还书、续借、遗失标记、预约创建、预约履约、罚款查询与缴纳等核心流程是否符合业务规则。

4. 边界与异常场景测试  
   验证未登录访问、参数缺失、无权限访问、资源不存在等情况下系统能否返回清晰的错误信息。

5. 多端兼容性验证  
   验证后端接口返回结构能否被 Web 端、Android 端和小程序端统一消费。

### 4.2 关键测试模块

为保证论文内容聚焦，本文不对全部接口逐一展开，而是选取最能体现系统业务价值的关键模块进行展示：

- 用户认证模块
- 图书借阅模块
- 预约排队模块
- 罚款管理模块
- 通知消息模块
- 管理员权限控制模块

## 6 自动化测试实现依据

本文测试并非停留在理论设计层面，而是直接依托项目中已实现的自动化测试代码。部分代表性测试文件如下：

- `backend-library/src/test/java/com/example/library/controller/AuthControllerTest.java`
- `backend-library/src/test/java/com/example/library/controller/LoanControllerTest.java`
- `backend-library/src/test/java/com/example/library/controller/ReservationControllerTest.java`
- `backend-library/src/test/java/com/example/library/controller/NotificationControllerTest.java`
- `backend-library/src/test/java/com/example/library/controller/FineControllerIntegrationTest.java`

其中：

1. `AuthControllerTest` 主要验证登录、登出、获取当前用户信息、刷新令牌和找回密码等接口。
2. `LoanControllerTest` 主要验证借阅记录查询、借书、还书、续借、标记遗失及权限控制。
3. `ReservationControllerTest` 主要验证预约创建、个人预约查询、管理员分页查询和预约履约操作。
4. `NotificationControllerTest` 主要验证通知列表获取、未读计数、单条已读、全部已读和删除已读通知。
5. `FineControllerIntegrationTest` 主要基于 H2 数据库验证罚款分页查询时的数据关联和返回结果。

以上测试代码能够从论文层面证明系统已经完成了较系统的接口验证，而非仅在界面层进行功能展示。

## 7 测试执行结果与分析

在本次测试中，选取用户认证、借阅管理、预约管理与罚款集成测试作为代表进行执行。实际执行命令如下：

```powershell
cd backend-library
.\mvnw.cmd "-Dtest=AuthControllerTest,LoanControllerTest,ReservationControllerTest,FineControllerIntegrationTest" test
```

执行结果显示：

- 共运行 35 个测试用例
- 失败数 0
- 错误数 0
- 构建结果为 `BUILD SUCCESS`

由此可以看出，系统在以下方面表现稳定：

1. 接口路由与请求映射正确。
2. 用户认证与权限校验机制能够正常工作。
3. 核心业务接口在正常场景与越权场景下均能返回符合预期的结果。
4. 涉及数据库实体关联的罚款查询流程能够在集成测试环境中稳定运行。

同时，测试过程中也发现系统仍存在一定可优化空间：

1. 前端自动化测试相对不足，目前测试重点主要集中在后端接口层。
2. 分页对象 `PageImpl` 的序列化存在框架层提示，后续可进一步封装统一分页响应结构。
3. 目前论文展示的联调结果仍以典型流程为主，若后续时间允许，可补充更多异常场景测试截图。

## 8 多端联调测试建议

由于本项目同时包含 Web 端、Android 端和小程序端，论文中可补充少量人工联调结果，以增强系统完整性说明。建议展示以下 3 组流程：

1. 登录联调  
   在三个客户端分别完成登录，验证均可成功获取用户身份信息和令牌。

2. 图书查询联调  
   在三个客户端输入相同关键字进行图书检索，验证结果数据来源一致。

3. 我的业务联调  
   在用户端查看我的借阅、我的预约、我的通知，验证三个端均可正确消费统一后端接口。

这种写法既能体现系统的多端适配能力，又不会让论文测试章节变成纯界面截图堆砌。

## 9 本章小结

本文围绕图书馆资源查询与借阅系统的核心业务，采用自动化接口测试与人工联调验证相结合的方式开展系统测试。测试结果表明，系统在认证鉴权、借阅管理、预约处理、罚款管理和通知消息等关键模块上运行稳定，能够满足预期业务需求。

相较于单纯的手工接口调试方式，自动化测试方案能够更系统地覆盖权限控制、异常分支与业务状态流转过程，也更适合作为毕业论文测试章节的支撑材料。后续若继续完善，可进一步补充前端自动化测试和更大规模的性能测试，以增强系统质量保障能力。
